当前代码里线程可以分成 5 类：主线程、每 core 的 dispatch 线程、每 core 的普通 executor worker、DT/VP 多实例 executor 线程、分布式通信辅助线程。

**1. 主线程**

入口通常是 `main_debug.py`，它创建 `NPUSimulator`，然后根据模式调用：

```
simulator.run()
simulator.run_parallel()
simulator.run_parallel_PC()
```

主线程的作用：

- 初始化 NPU、core、memory、trace、instruction。
- 启动每个 core 的 dispatch 线程。
- `join()` 等待各 core dispatch 线程结束。
- 检查 worker 线程是否报错。
- dump memory 和 trace。

在 `run_parallel()` 里：

```
core.instruction_dispatch_class.dispatch_thread.start()
core.instruction_dispatch_class.dispatch_thread.join()
```

在 `run_parallel_PC()` 里：

```
core.instruction_dispatch_class.dispatch_thread_PC.start()
core.instruction_dispatch_class.dispatch_thread_PC.join()
```

**2. 每个 core 的 dispatch 线程**

每个 core 有两个 dispatch thread 对象，但一次运行只用其中一个：

```
self.dispatch_thread
self.dispatch_thread_PC
```

它们定义在 [instruction_dispatch.py (line 218)](/home/zhangds/Zeus3FunctionalSimulator/npu_core_simulator/instruction_dispatch/instruction_dispatch.py:218)。

作用：

- 按 PC 或指令列表顺序读取本 core 的 instruction。
- 调 `parse_to_dict()` 解析字段。
- 对 CT / SET 类指令直接执行。
- 对 LD/LW/ST/DT/VP/PE/SI/SO 计算类指令，生成 `latch_data`，放入对应 executor queue。
- 更新 PC。
- 等待所有 executor queue 完成。
- 保存 per-core trace。

普通 parallel 模式用：

```
instruction_dispatch()
```

parallel-pc 模式用：

```
instruction_dispatch_PC()
```

区别：

- `instruction_dispatch()` 用 for-loop 顺序遍历指令，并把 `core_regs.PC` 设成当前 index。
- `instruction_dispatch_PC()` 用真实 `PC` 循环，CT jump 等改 PC 的行为可以生效。
- parallel-pc 还会给每条指令标注 `(PC, core_index)`，用于 deterministic gate 保证跨 core 提交顺序稳定。

**3. 每个 core 的普通 executor worker 线程**

每个 core 初始化时会启动这些 daemon worker：

```
LD_worker
LW_worker
DT_worker
ST_worker
VP_worker
PE_worker
SI_worker
SO_worker
```

对应队列：

```
self.queues = {
    'LD': Queue(),
    'LW': Queue(),
    'DT': Queue(),
    'ST': Queue(),
    'VP': Queue(),
    'PE': Queue(),
    'SI': Queue(),
    'SO': Queue(),
}
```

作用关系是：

```
dispatch thread
  -> _enqueue_ld/_enqueue_st/...
  -> queues['LD']/queues['ST']/...
  -> 对应 worker thread
  -> executor class
```

例如 LD：

```
dispatch_thread
  -> queues['LD'].put(...)
  -> LD_worker
  -> LD_excuter_class.LD_excuter()
```

这些 worker 的共同流程：

1. 从自己的 queue 取指令。
2. 等待 mail 依赖满足：

```
等待发射条件(...)
```

1. 记录 trace begin。
2. 调具体 executor。
3. 记录 trace end。
4. `task_done()`。
5. `notify()` 唤醒可能因队列长度阻塞的 dispatch 线程。

其中 CT 指令没有单独 worker。CT 在 dispatch 线程里同步执行，因为 CT 会改寄存器、PC、mail、配置等，很多行为需要保持发射顺序。

**4. DT / VP 多实例 executor 线程**

DT 和 VP 比较特殊，它们是两级线程模型。

第一层是主 worker：

```
DT_worker
VP_worker
```

第二层是多个真实执行实例线程：

```
DT0_exec, DT1_exec, ...
VP0_exec, VP1_exec, ...
```

创建位置：

```
self.dt_queues = [Queue() for _ in range(core.num_dt_executors)]
self.dt_threads = [...]

self.vp_queues = [Queue() for _ in range(core.num_vp_executors)]
self.vp_threads = [...]
```

默认数量来自：

```
num_dt_executors = SIMU_PARAMS.get("num_dt_executors", 4)
num_vp_executors = SIMU_PARAMS.get("num_vp_executors", 4)
```

DT 流程：

```
dispatch thread
  -> queues['DT']
  -> DT_worker
      - 等待 Rec Mail
      - 消费 Rec Mail
      - 根据 CSS 路由到 DT0/DT1/...
      - 放入 dt_queues[dt_id]
  -> DTx_exec thread
      - 真正执行 DT_TRA / DT_RES
      - 完成后通过 retire gate 发送 Send Mail
```

VP 流程类似：

```
dispatch thread
  -> queues['VP']
  -> VP_worker
      - 等待 Rec Mail
      - 根据 CSS 路由到 VP0/VP1/...
      - 放入 vp_queues[vp_id]
  -> VPx_exec thread
      - 真正执行 VP_STSR / VP_DTSR / VP_SET*
```

为什么要两级：

- `DT_worker` / `VP_worker` 负责程序顺序相关的 mail 判断、mail 消费、CSS 路由。
- `DTx_exec` / `VPx_exec` 模拟多个并行执行单元。
- `_RetireGate` 保证多个实例乱序完成时，Send Mail 仍按程序顺序释放。

**5. Mailbox 和同步关系**

每个 core 有一个 mailbox 矩阵：

```
self.mailbox[receiver][sender]
```

例如：

```
mailbox['DT']['LD']
```

表示 LD 给 DT 发的 token 数。

线程关系：

- producer executor 完成后调用 `指令结束寄信()`，增加 mailbox token。
- consumer worker 发射前调用 `等待发射条件()` 等 token。
- 真正开始执行时调用 `指令开始收信()` 消费 token。

为了线程安全，mailbox 由：

```
self.mailbox_lock
```

保护。

DT/VP 的一个重要细节是：
它们的 Rec Mail 在 `DT_worker` / `VP_worker` 中消费，而不是在 `DTx_exec` / `VPx_exec` 中消费。这样避免多个实例线程同时 judge/consume 造成依赖竞态。

**6. parallel-pc deterministic gate**

`run_parallel_PC()` 会打开：

```
self.parallel_pc_deterministic_gate.enabled = True
```

这个 gate 是 NPU 级对象：

```
ParallelPCDeterministicGate
```

作用：

- 多 core、多 worker 线程实际调度顺序由 OS 决定，可能不稳定。
- parallel-pc 模式下，为每条指令分配稳定 key：

```
(PC, core_index)
```

- worker 在提交关键副作用时进入 gate。
- gate 保证低 PC、低 core_index 的指令先提交。

它不是执行功能必需的硬件模型，更像是仿真器为了让并行执行结果稳定、可复现而加的排序机制。

部分指令会绕过这个 gate，例如：

- `CT_WAIT`
- `CT_SEND`
- `*_MAIL`
- `SI/SO`

因为这些本身就是同步/通信类行为，强行按 PC gate 可能造成假死或破坏同步语义。

**7. 错误传播线程关系**

所有 worker 和 dispatch thread 都通过 `_thread_guard()` 包装。

如果某个 worker 抛异常：

1. 记录到 NPU 级：

```
self.thread_exceptions
```

1. 设置：

```
self.stop_event.set()
```

1. 其他 dispatch loop 看到 `stop_event` 后退出。
2. 主线程 join 后调用 `_raise_if_any_thread_failed()`，把异常重新抛回主线程。

这避免了 Python daemon worker 静默失败，主线程还误以为仿真成功。

**8. Memory conflict / timing 相关线程机制**

多个 executor worker 会并发访问：

- 本 core L2
- die 共享 GDG
- core 独享 Weight DRAM

所以代码里有：

- `MemoryConflictTracker`
- `memory_region_register_lock`
- executor 级 `_register_memory_access()`
- `timing_policy.defer_writes()`

`defer_writes()` 用的是：

```
threading.local()
```

也就是每个 worker 线程有自己的 pending write 列表。这样多个 executor 同时进入 middle timing policy 时，不会互相覆盖 pending write 状态。

**9. CT_WAIT / CT_SEND 的跨 core 同步线程关系**

每个 core 有：

```
self.sync_counter = [0] * 9
self.sync_cond = threading.Condition()
```

`CT_SEND` 会增加目标 core 的 `sync_counter`，并 `notify_all()`。
`CT_WAIT` 会在 `sync_cond` 上等待对应 counter 大于 0，然后消费。

这是跨 core 同步，不通过 executor queue，而是直接作用在目标 core 的 sync 状态上。

**10. 分布式模式里的额外线程**

常规 `main_debug.py` 本地仿真主要用上面那些线程。代码里还有分布式/多进程支持，涉及额外线程：

- ```
  InboundServer
  ```

  - 每 core process 内一个 daemon thread。
  - 负责接收远端 L2 read/write、SIGNAL、SYNC 消息。

- ```
  PipeTransport
  ```

  - 有一个 background reader thread。
  - 负责从 pipe 读消息，按 `req_id` 分发到 pending request 或 inbox。

- ```
  LocalBroker
  ```

  - 每个 worker connection 一个 pump thread。
  - 负责在 worker 之间转发消息。

这些属于 distributed/wire 路径，不是普通单进程 debug case 的主线程模型。

**整体关系图**

```
主线程
  |
  |-- NPUSimulator 初始化
  |     |
  |     |-- 每个 core 创建 instruction_dispatch_class
  |             |
  |             |-- 启动 daemon workers:
  |             |     LD_worker
  |             |     LW_worker
  |             |     DT_worker ----> DT0_exec / DT1_exec / ...
  |             |     ST_worker
  |             |     VP_worker ----> VP0_exec / VP1_exec / ...
  |             |     PE_worker
  |             |     SI_worker
  |             |     SO_worker
  |
  |-- run_parallel / run_parallel_PC
        |
        |-- 每个 core 启动一个 dispatch thread
        |       |
        |       |-- CT / SET 类：dispatch thread 内直接执行
        |       |
        |       |-- LD/LW/ST/PE/SI/SO：放入对应 queue
        |       |
        |       |-- DT/VP：放入主 queue，再由 DT_worker/VP_worker 路由到多实例执行线程
        |
        |-- 主线程 join 所有 dispatch thread
        |
        |-- 检查 worker 异常 / mailbox / dump trace
```

**一句话总结**

当前仿真器是“每 core 一个 dispatch 线程 + 多个 executor worker 线程”的模型；CT 和配置类指令在 dispatch 线程中保持顺序执行，数据搬运和计算类指令进入各自 executor queue 并发执行；DT/VP 额外支持多实例执行线程，通过 CSS 路由和 retire gate 保证并行执行下的 mail 顺序。
