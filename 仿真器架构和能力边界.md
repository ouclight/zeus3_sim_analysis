> 两个版本当前是职责互补，不是简单的新旧替代关系。

## 第8页：指令执行与调度模型

### 建议内容

执行单元包括：

| 类别  | 主要职责                             |
| ----- | ------------------------------------ |
| CT    | 控制流、跳转、循环、寄存器配置、同步 |
| LD    | 从DDR或其他空间加载数据到L2          |
| LW    | 加载权重或相关数据                   |
| ST    | 将L2数据写回目标存储空间             |
| DT    | 数据搬运、重排或格式相关处理         |
| VP    | 向量计算、归约、广播、LUT等          |
| PE    | 矩阵乘、卷积、累加等主要计算         |
| SI/SO | 流输入输出和执行单元之间的数据连接   |

### Python调度

- `serial`：按指令顺序执行，适合快速功能验证；
- `parallel`：执行单元采用并发 worker；
- `parallel-pc`：CT 与多个 executor 并行推进，更接近硬件并发语义。

### C++调度

- 单线程确定性事件模型；
- 各 executor 独立维护 busy、ready、commit 状态；
- mailbox 和 barrier 控制可发射性；
- 空闲周期可以 fast-forward；
- trace-only 校准模式支持多条指令处于 in-flight 状态。

边界提示：

> “并行执行”表示仿真器对硬件并发关系的抽象，不代表宿主机线程数，也不意味着完全还原 RTL 流水线。

## 第10页：同步、Mailbox与多核执行

### 页面内容

仿真器对以下同步关系进行建模：

- executor send mail / receive mail；
- 同一执行 lane 的 FIFO 顺序；
- CT_SEND / CT_WAIT；
- SISO SO→SI transfer；
- Core 之间的 token；
- 显式依赖边；
- barrier 和 drain 语义。

### 建议图示

```
Core 0                              Core 1
 CT                                  CT
 │                                   │
 ├── dispatch PE                     ├── CT_WAIT
 │        │                          │      ▲
 │        ▼                          │      │
 │       PE ── send mail ──► mailbox│      │
 │                                   │      │
 └── CT_SEND ───────────────► token bank ──┘
```

### 边界说明

- 仿真器能判断“已建模的同步是否满足”；
- 能发现 mail 丢失、脏 mailbox、等待超时等问题；
- 能记录本次动态执行中的 happens-before 关系；
- 不能证明所有硬件隐含顺序均已建模；
- 未记录的硬件仲裁或隐式屏障可能造成误报；
- 也可能因 footprint 漏记导致漏报。

## 第11页：Trace与诊断体系

### 建议分层说明

| 诊断输出            | 用途                                 |
| ------------------- | ------------------------------------ |
| NPU级Trace          | 查看多Core整体执行关系               |
| Core级Trace         | 查看PC、CT和各执行单元时序           |
| Executor Trace      | 查看具体指令开始、结束、阻塞和提交   |
| Register状态        | 定位控制流和参数错误                 |
| Memory Dump         | 定位数据搬运和计算错误               |
| Golden Compare      | 判断最终输出是否符合预期             |
| Race Detector       | 观察Python并发执行中的实际抢跑       |
| Dependency Analyzer | 判断memory overlap是否具备可证明顺序 |
| Timing Stress       | 用不同cost组合暴露隐藏时序依赖       |

### 重要说明

Trace回答的是：

- 执行了什么；
- 以什么顺序执行；
- 为什么阻塞；
- 访问了哪些地址；
- 何时产生或消费同步；
- 哪一步首次偏离预期。

Trace不能自动回答：

- RTL信号为什么翻转；
- 实际芯片为什么慢；
- 某个性能差异是否来自DDR、NoC或流水线；
- 编译器应该采用哪种最优调度。

## 第12页：Dependency Conflict Analysis原理

### 建议图示

```
动态指令执行
      │
      ├── memory access: resource + [start, end) + R/W
      └── dependency edge: FIFO / mail / CT sync / SISO
                         │
                         ▼
                 Happens-Before DAG
                         │
                         ▼
              按资源做地址区间重叠扫描
                         │
             ┌───────────┴───────────┐
             ▼                       ▼
       有可达顺序                 无可达顺序
          安全                 RAW / WAR / WAW
```

### 能做什么

- 记录本次动态执行的 memory access；
- 将远端地址归一化到实际 owner；
- 识别同一资源的重叠地址范围；
- 根据依赖图检查访问是否有序；
- 报告 unordered RAW、WAR、WAW；
- 作为 CI failure gate。

### 不能做什么

- 不是静态程序证明；
- 不覆盖未执行分支；
- 不分析 read-read；
- 不判断数据值是否正确；
- 不检测 bank、cache、capacity conflict；
- 不判断真实 wall-clock 是否发生重叠；
- 不等同于 RTL hazard detector；
- 当前不能完整合并所有跨 launch 的全局依赖。

- 支持部分多Device和远端访问；

### 4. Runtime集成验证

- device注册；

## 第15页：仿真器不能做什么

建议将本页设计为红色或灰色边界声明页。

### 1. 不能替代RTL仿真

当前仿真器不提供：

- RTL信号级波形；
- 组合逻辑行为；
- 真实流水寄存器状态；
- CDC、复位、时钟门控；
- X/Z传播；
- setup/hold；
- 信号级仲裁；
- RTL实现缺陷的普遍复现。

个别已知RTL缺陷可以做定向模拟，但不代表整体RTL等价。

### 2. 不能默认提供Cycle-accurate性能

仿真器可以输出抽象tick和可配置cost，但默认不能保证：

- 总周期与芯片一致；
- 单指令latency与RTL一致；
- executor吞吐与真实硬件一致；
- DDR、L2和fabric拥塞一致；
- 多核竞争和仲裁一致。

只有完成硬件校准并经过误差验证的模型，才可用于性能评估。

### 3. 不能证明程序对所有输入正确

仿真器执行的是具体输入下的动态路径，因此不能证明：

- 未执行分支正确；
- 所有loop边界正确；
- 所有shape正确；
- 所有地址组合都不会越界；
- 所有并发交错都安全。

### 4. 不能替代编译器静态检查

仿真器可以发现错误结果，但通常不能自动判断：

- 错误来自哪个编译Pass；
- schedule为什么不合法；
- 哪个IR变换破坏了语义；
- 应该如何修复编译器；
- 哪一种tiling最优。

### 5. 不能覆盖所有系统和物理行为

目前不完整覆盖：

- 功耗和温度；
- ECC和软错误；
- Cache/L2 bank冲突；
- DDR控制器细节；
- NoC拥塞；
- 中断和操作系统行为；
- 多进程抢占；
- 芯片封装和物理限制。

## 第16页：结果应该如何解读

这页非常重要，可以防止使用者“过度相信”仿真结果。

| 仿真结果              | 可以得出的结论               | 不能直接得出的结论   |
| --------------------- | ---------------------------- | -------------------- |
| Golden compare通过    | 当前输入和当前模型下输出正确 | RTL和芯片一定正确    |
| Golden compare失败    | 仿真结果与golden不一致       | 一定是编译器错误     |
| 无dependency conflict | 已记录访问在当前依赖图中有序 | 所有并发场景都安全   |
| 报告RAW/WAR/WAW       | 模型无法证明两次访问有序     | 硬件一定会算错       |
| Trace tick更短        | 当前cost模型估计更短         | 芯片真实性能一定更快 |
| Python和C++一致       | 两个模型在该case上结果一致   | 两边都符合RTL        |
| 500次回归通过         | 在500次采样中未复现问题      | 不存在低概率竞态     |
| Analysis-only通过     | 控制路径和依赖分析完成       | Tensor数值正确       |

建议在页面底部加入一句：

> 仿真结论的可信度，由“ISA语义正确性、模型完整性、输入覆盖率和硬件校准程度”共同决定。

## 第17页：当前已知限制与风险

建议实事求是列出，不宜隐藏。

### ISA和语义对齐

- Python和C++正在通过静态审计及差分测试持续对齐；
- 当前静态审计覆盖151条mnemonic；
- C++识别148条，Python识别147条；
- 仍有少量双方未覆盖的指令定义；
- 静态“已覆盖”不等价于所有字段组合均已验证。

### Python版风险

- `parallel-pc`受线程调度、GIL和BLAS线程影响；
- 可能出现低概率并发问题；
- wall-clock和sleep不适合作为稳定性能依据；
- JSON trace可能占用较大内存和时间。

### C++版风险

- 主要运行环境是Linux/WSL；
- daemon和共享内存依赖POSIX能力；
- C++诊断便利性暂时弱于Python；
- 默认timing cost尚未完成硬件校准；
- 在线race detector能力弱于Python；
- 跨Device的全局dependency/timing仍有边界。

## 第19页：演进方向

建议把优化方向分成“可信度、性能、集成”三条线。

### 可信度

- 统一ISA schema；
- 自动生成字段和译码结构；
- 建立Python/C++单指令差分；
- 建立kernel lockstep；
- 补齐已知ISA gap；
- 加强数值边界测试；
- 建立RTL/芯片golden回灌。

### 性能

- 完善事件驱动fast-forward；
- 减少热路径字符串和动态分配；
- trace分级与流式输出；
- Tensor scratch arena；
- SIMD BF16/FP8转换；
- program decode cache；
- resident session和memory snapshot。

### 性能模型与系统集成

- 使用RTL/芯片数据校准cost；
- 增加L2 bank和DDR带宽模型；
- 增加fabric contention；
- 建设跨Device全局事件模型；
- 稳定Runtime ABI；
- 提供编译器直接调用的library API。

## 第20页：总结

建议只保留四句话：

> 1. ZEUS-V3仿真器是连接编译器与硬件验证的重要软件模型。  
> 2. Python侧重参考模型和调试，C++侧重确定性、吞吐和Runtime集成。  
> 3. 仿真器能够验证指令功能、数据结果、同步关系和动态依赖，但不等价于RTL和芯片。  
> 4. 使用仿真结果时，必须区分功能正确性、依赖正确性和性能准确性。

结尾可以使用一句正式的边界声明：

> 仿真器的价值不是“代替硬件”，而是在硬件之前，以更低成本、更高可观测性发现编译器、指令流和运行时集成问题。

# 四、建议在PPT中重点突出三组边界

## 1. 功能正确性与性能正确性的边界

```
Functional模式通过
        ≠
Cycle数量准确
        ≠
真实芯片性能准确
```

功能模型回答“结果是什么”，性能模型回答“需要多久”。二者必须分开验收。

## 2. 动态验证与静态证明的边界

```
当前输入、当前路径没有发现问题
        ≠
所有输入、所有路径都没有问题
```

仿真器是动态执行工具，不是形式化验证工具。

## 3. 抽象硬件语义与RTL实现的边界

```
符合ISA抽象语义
        ≠
完全复现RTL微架构行为
```

仿真器可以发现编译器和指令语义问题，但不能覆盖所有RTL实现缺陷。

整份PPT建议使用三类视觉元素：

- 架构页：分层方框图；
- 运行机制页：时序图或数据流图；
- 边界页：能力矩阵和“能/不能”双栏。

颜色可以固定为：

- 蓝色：公共架构和功能能力；
- 绿色：Python实现；
- 橙色：C++实现；
- 红色：限制和禁止外推的结论；
- 灰色：RTL、芯片等外部系统。

避免大段源代码，最多保留少量接口名。汇报对象通常关心的是职责、数据流和可信边界，而不是具体类名。

如果汇报时间不足，可以压缩成12页：保留第2、3、4、5/6合并、7、8、9、11/12合并、14、15、16、20页。
